Introduction

本次实验的目的是为我们的OS编写内存管理代码，分为两部分：

  1. physical memory allocator：维持一个数据结构，记录物理页的分配情况、共享情况，并允许分配；注意一个页的大小是4096bytes。

  2. virtual memory manager：将虚拟地址map到物理地址。
    使用命令创建并切换分支到lab2，以后的实验里可以不带参数b切换分支。

    本实验需要重点关注新提供的几个文件：
    inc/memlayout.h：struct PageInfo保存了物理页的metadata，与物理页有一一对应的关系，可以利用pp_link找到下一个空闲界面；。

    ，尤其是inc/memlayout.h和kern/pmap.h；后者定义了PageInfo structure，可用于追踪空闲的物理页。

Part1 物理页管理

实现我们的physical page allocator（的一部分）。

Exercise1

在文件kern/pmap.c中实现下面的函数boot_alloc()、mem_init()、page_init()、page_alloc()、page_free()。只需要在mem_init()中完成check_page_list()的测试即可。
阅读xv6手册chap2，我们知道了xv6的虚拟地址究竟如何通过页表机制映射为物理地址。页表以tow-level tree的形式存储在物理内存中，其根为4096-byte page directory，连接1024个页表页（page tabe pages），每个页表都是1024个32bit的array。其中每个虚拟地址的前10位用于查询page directory，如果entry存在，便用接下来的10位查询对应的oage table，最后的12位作为偏移保留，如此便得到了最后的20位物理地址。mmu.h中有虚拟地址的具体定义。此外这是多级页表的一种典型实现。
每个进程都有自己的two-level 页表，所以我们需要一个allocator在运行时动态分配物理内存。我们的allocator有一个链表page_free_list来追踪空闲的物理页，它是一个栈结构，使用page_alloc（）出栈，page_free()入栈，但是在刚开机时，这个链表尚未初始化，我们无法部署内核自身的二级页表。解决方法是额外使用boot_alloc（）分配，在page_init()建立好内核二级页表和page_free_list后就用page_alloc（）代替boot_alloc（）。

boot_alloc()：只用于JOS初始化时的物理内存分配。其中ROUNDUP是一个用于对齐的宏，这给我们写代码提供了很大的方便。

mem_init()：用于设置JOS的two-level页表。Excercise1中需要实现的部分比较少，只要分配二级页表的物理存储空间即可。

page_init()：初始化空闲页链表，自此可以使用page_*()进行内存管理。首先来了解一下struct PageInfo的定义：

从中我们可以知道对空闲页和忙碌页应该如何做不同的设置，另一个需要关注的就是哪些页已经被占用为忙碌页：

  1. page 0：存储IDT和BIOS structure的重要页；
  2. IO洞：[IOPHYSMEM, EXTPHYSMEM)这部分留给I/O，不可以用于分配；

  3. boot_alloc：我们之前分配给JOS的二级页表处在[EXTPHYSMEM,PADDR(nextfree))，这部分无法回收，无法分配；此外对于如何得到nextfree有一个巧妙地办法。

    page_alloc()：分配一个物理页，输入的条件决定是否要对分配的物理页清零初始化。

    page_free()：牺牲一个物理页，要确认清楚的页已经没有人关心了（被引用量为0）。

    综上，我们初步认识了JOS如何实现分页。

Part2 虚拟内存

首先需要熟悉x86的保护模式内存管理体系——分段&分页

Exercise2

阅读Intel 80386 Reference Manual的chap5和chap6，学习一下分段分页机制和保护机制。
阅读资料，我们知道了：

  • 段描述符提供了将逻辑地址映射为线性地址的数据，段描述符不是由应用程序创建的。
  • 段描述符存储在描述符表中：GDT（全局描述符表）、LDT（局部描述符表）；其中GDT的第一个条目不被进程使用，指向两个表的指针分别是GDTR、LDTR。
  • 使用逻辑地址中的selector部选择一个描述符表并indexing一个描述符。
  • 段寄存器用于存储描述符中得来的信息，它的前16位是可见的，用于存储selectors。
  • 页表本身也是页，将线性地址分成10*2+12位，前20位用二级页表索引，如此可以索引2^32位地址。
  • 保护模式下，分段分页的条目中很多有很多具有保护机制的位。 上图中可见，用寄存器CR3保存了page directory的首址。

对于CPU执行的指令，一旦处在保护模式，就无法直接接触线性地址或物理地址；即C指针的值全都是虚拟地址。对指针的解引用需要经过virtual-linear-physical的映射，所以直接对物理地址解引用是不可行的，下面给出几种指针类型及其对应的地址类型

uintptr_t virtual
physaddr_t physical

页表管理

编写能够管理页表，插入删除linear-physical地址映射，创建页表页的程序。

Exercise4

实现kern/pmap.c中的pgdir_walk()、boot_map_region()、page_lookup()、page_remove()、page_insert()，使用mem_init()中调用的check_page进行测试
设置二级页表的时候要注意，用PTE_ADDR取PDE和PTE都得到物理地址，其中不能直接对PDE的结果解引用来设置PTE（见上面对指针类型的讨论），而应该先使用KADDR()来得到内核虚拟地址才能解引用设置PTE。
pgdir_walk()：从二级页表中找到线性地址va对应的PTE并返回其指针，如果没有page table，就创建并返回一个页表页。注释有提醒我们将PDE设置得permissive一些，但是没想到在检查的断言中才发现居然是居然至少得是是PUW。

boot_map_region()：设置UTOP上面的一些“静态”mapping，这部分属于不会释放的内核的页，不需要对这里的物理页进行计数。

page_lookup()：找到map到va的物理页，可用于page_remove()，需要注意没有pte和pte里没有物理页两种无效情况。

page_remove()：取消物理页在虚拟地址va的映射，如果va本来就没有物理页，就静静地什么也不做。注意在给PTE清零时，需要废除TLB。tlb_invalidate()函数能够作废TLB，TLB似乎是用于快速缓存页表，减少查询时间的函数。

page_insert()：物理页pp映射到虚拟地址va，如果已经有页映射到va了，就应该移除这个页面。其中注释额外提醒如果已经映射到va上的页和现在提供的页pp是同一个页，需要特别关注会发生什么。注释中提到可以使用一种优雅的方法不单独处理这种情况，我暂时还没想到。
